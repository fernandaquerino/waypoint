# ADR-004: Autenticacao com Auth.js, Drizzle e NestJS

**Status:** aceito  
**Data:** 2026-07-07  
**Contexto:** definir login, sessao e validacao de identidade em Next.js + NestJS + PostgreSQL.

## Contexto

O Waypoint precisa de login real desde a Sprint 0 para que todas as entidades futuras nascam isoladas por usuario. A arquitetura do produto usa Next.js como camada web/BFF, NestJS como API de regras de negocio e PostgreSQL como banco principal.

Requisitos desta decisao:

- login com Google;
- login e cadastro com email/senha;
- senha salva apenas como hash;
- sessao persistente em cookie httpOnly gerenciado pelo Auth.js;
- JWT validado pelo NestJS em rotas protegidas;
- `userId` sempre derivado da sessao/token, nunca do client.

## Decisao

Usaremos **Auth.js no Next.js** como autoridade de autenticacao e **NestJS validando JWT assinado** nas rotas protegidas.

O Auth.js roda no App Router do Next.js com:

- provider Google;
- provider Credentials para email/senha;
- `@auth/drizzle-adapter` usando os schemas Drizzle de `@waypoint/db`;
- estrategia de sessao `jwt`;
- encode/decode customizado em JWS HS256 para o NestJS validar offline.

O NestJS expoe endpoints de credenciais e uma rota protegida:

- `POST /auth/register`;
- `POST /auth/login`;
- `GET /auth/me`.

O endpoint `GET /auth/me` usa guard JWT e retorna apenas o `userId` extraido do token validado.

## Segredo Compartilhado

A decisao e usar **somente `AUTH_SECRET`** como segredo compartilhado.

`AUTH_SECRET`:

- assina o JWT no Auth.js;
- valida o JWT no NestJS;
- deve ser o mesmo valor para web e API;
- substitui a necessidade de um `JWT_SECRET` separado nesta fase.

Variaveis locais:

```env
AUTH_SECRET=
AUTH_URL=http://localhost:3000
API_URL=http://localhost:3333
WEB_URL=http://localhost:3000
DATABASE_URL=postgresql://waypoint:waypoint@localhost:5432/waypoint
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
```

## Persistencia

O schema de autenticacao fica em `packages/db`:

- `users`: usuario base, email unico, imagem, timestamps, `password_hash` nullable e `email_verified`;
- `accounts`: vinculo OAuth usado pelo Auth.js para Google.

`password_hash` e nullable porque usuarios criados por Google nao precisam ter senha local.

## Fluxos

Cadastro por email/senha:

1. Next.js chama `POST /auth/register`.
2. NestJS normaliza email, gera hash com bcrypt e persiste no Drizzle.
3. Next.js chama `signIn("credentials")`.
4. Auth.js emite cookie de sessao com JWT assinado.

Login por email/senha:

1. Auth.js Credentials chama `POST /auth/login`.
2. NestJS compara senha com o hash salvo.
3. Auth.js emite sessao/JWT quando o usuario e valido.

Login com Google:

1. Auth.js executa o fluxo OAuth.
2. `@auth/drizzle-adapter` cria ou reutiliza `users` e `accounts`.
3. Auth.js emite sessao/JWT com `sub = users.id`.

Chamada Web -> API protegida:

1. Next.js le o JWT do cookie httpOnly no servidor.
2. Next.js chama a API com `Authorization: Bearer <token>`.
3. NestJS valida o token com `AUTH_SECRET`.
4. Controllers usam apenas `req.user.id`.

## Consequencias

Positivas:

- OAuth fica com Auth.js, sem reimplementar Google manualmente.
- NestJS treina e usa guard JWT real.
- O browser nao armazena token em `localStorage`.
- O backend valida identidade sem confiar em `userId` enviado pelo client.

Trade-offs:

- `AUTH_SECRET` precisa estar sincronizado entre web e API.
- JWT assinado nao tem revogacao instantanea antes de expirar.
- O formato customizado HS256 deve ser revisado em upgrades grandes do Auth.js.

## Fora da Decisao

- recuperacao de senha;
- verificacao de email;
- RBAC;
- 2FA;
- exclusao completa de conta.
