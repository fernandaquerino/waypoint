# ADR-004: Autenticação entre Auth.js (Next.js) e NestJS

**Status:** proposto
**Data:** 2026-07-07
**Contexto:** definir estratégia de autenticação em arquitetura Next.js + NestJS + Postgres

## Contexto

Arquitetura decidida: browser fala com Next.js, Next.js fala com NestJS, NestJS fala com Postgres. Next.js atua como BFF, NestJS como backend principal.

Requisitos:

- Login com email/senha na v1. OAuth (Google, GitHub) opcional pra depois
- Sessão persistente em cookie httpOnly (nada de token no localStorage)
- Guards no NestJS validando identidade em todo endpoint autenticado
- SSR do Next precisa saber quem tá logado antes de renderizar
- Fluxo deve funcionar em desenvolvimento local (Docker compose) e produção

Restrição de contexto:

- Você já usa **Auth.js v5** no FinSight AI. Reaproveitar essa fluência é ganho real
- Você quer **aprender NestJS deeply**, inclusive guards e Passport
- Side project 2h/dia: nada de reimplementar OAuth do zero

## Opções consideradas

### Opção A: Auth 100% no NestJS (Passport + estratégias)

NestJS assume tudo. Passport-local pra email/senha, passport-google-oauth20 pra Google. Sessão em cookie assinado pelo NestJS. Next.js é só cliente.

**Prós:**

- Uma única fonte de verdade pra auth
- NestJS "puro" com Passport, aprende o padrão canônico da comunidade

**Contras:**

- Você reimplementa fluxos que Auth.js já resolve (OAuth callback, PKCE, state, verificação de email)
- Dobra o trabalho por provider adicional
- Perde a fluência que você já tem em Auth.js
- CORS entre browser e NestJS vira dor de cabeça se um dia expuser o NestJS direto

### Opção B: Auth.js como autoridade + JWT compartilhado com NestJS

Auth.js no Next.js gerencia login (credentials, OAuth), emite JWT com sessão. Next.js server-side inclui esse JWT no header quando chama NestJS. NestJS valida usando o **mesmo secret**.

**Prós:**

- Aproveita Auth.js pra OAuth complicado (Google, GitHub, magic link, etc.)
- Você aprende `@nestjs/passport` + `passport-jwt`, que é o padrão real de guards em produção
- Sem reimplementação: o difícil (state, PKCE, tokens de provider) fica com Auth.js
- Cookie do Auth.js é httpOnly + SameSite=Lax por padrão

**Contras:**

- Requer disciplina com o `JWT_SECRET` (mesma variável nos dois apps)
- Formato do JWT precisa ser compatível entre Auth.js e Passport (HS256, claims esperados)
- Rotação de secret exige coordenação

### Opção C: Auth.js + sessão opaca + callback de validação

Auth.js emite cookie de sessão. NestJS não decodifica; faz callback pro Next (`/api/auth/validate`) pra saber quem é o usuário.

**Prós:**

- Sem compartilhamento de secret
- NestJS não precisa entender formato interno da sessão

**Contras:**

- Toda requisição autenticada no NestJS vira duas requisições (NestJS → Next → resposta)
- Latência dobra
- Se Next cair, backend fica sem auth
- Padrão pouco comum, você improvisa muito

### Opção D: BFF proxy total (browser nunca fala com NestJS)

Todas as chamadas do browser vão pra Next.js (API routes ou Server Actions). Next.js chama NestJS server-to-server com token machine-to-machine + user_id no header.

**Prós:**

- NestJS nunca é exposto na internet
- Sem CORS
- Auth do browser é responsabilidade só do Next

**Contras:**

- Duplica caminho de request (browser → Next → NestJS) mesmo pra queries triviais
- Server Actions do Next viram wrapper burocrático pra cada endpoint do NestJS
- Você perde a chance de treinar guards de verdade no NestJS, porque a autoridade tá no Next

## Decisão

**Opção B, combinada com um recorte da Opção D.**

Auth.js é a autoridade de identidade (login, cadastro, OAuth futuro). NestJS valida JWT emitido pelo Auth.js via `@nestjs/passport` + `passport-jwt`. Guards do NestJS protegem endpoints como qualquer API real.

Do lado do browser, adotamos o BFF parcialmente: **mutations e queries sensíveis passam pelo Next** (Server Actions e Route Handlers) que forwardam pro NestJS com o JWT. Queries públicas ou GETs simples de client components **podem chamar o NestJS direto**, com o JWT vindo do cookie via um proxy leve no Next.

O resultado híbrido dá o melhor dos dois mundos: você aprende guards no NestJS de verdade, e a fronteira pra internet é o Next.

## Consequências

**Positivas:**

- Você reusa Auth.js sem reinventar OAuth
- Aprende `@nestjs/passport`, `passport-jwt`, `@UseGuards`, decorators customizados (`@CurrentUser`) — que é o padrão real
- Cookie httpOnly gerenciado pelo Auth.js: sem token no localStorage, sem XSS trivial
- JWT verificável offline pelo NestJS: nada de callback extra por request

**Negativas:**

- Você precisa configurar Auth.js pra usar HS256 e claim `sub = user.id`. Não é o default absoluto, mas é uma linha de config
- `AUTH_SECRET` (Auth.js) e `JWT_SECRET` (NestJS) precisam ser o mesmo valor. Se divergirem, tudo quebra com erro genérico de "unauthorized"
- Refresh de token: Auth.js reemite JWT a cada 30 dias por default. Enquanto o token é válido, revogação instantânea não existe. Aceitável pra v1

**Neutras:**

- Precisa validar formato do JWT em ambos apps quando fizer upgrade de Auth.js ou passport-jwt
- Logout no Next apaga o cookie. NestJS não sabe disso, mas como o token não vai mais junto no header, na prática funciona

## Detalhes de implementação

### Variáveis de ambiente compartilhadas

`.env` na raiz do monorepo, lido por Docker compose e pelos dois apps:

```
AUTH_SECRET=<gerado com openssl rand -base64 32>
DATABASE_URL=postgresql://waypoint:waypoint@postgres:5432/waypoint
API_URL=http://localhost:3001
WEB_URL=http://localhost:3000
```

Regra: `AUTH_SECRET` é usado por Auth.js pra assinar E pelo NestJS pra verificar. Nomeamos como `AUTH_SECRET` porque a autoridade é o Auth.js.

### Configuração do Auth.js (Next.js)

`apps/web/src/auth.ts`:

```typescript
import NextAuth from "next-auth";
import Credentials from "next-auth/providers/credentials";
import { z } from "zod";

const credentialsSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
});

export const { auth, handlers, signIn, signOut } = NextAuth({
  session: { strategy: "jwt", maxAge: 30 * 24 * 60 * 60 }, // 30 dias
  secret: process.env.AUTH_SECRET,
  providers: [
    Credentials({
      credentials: {
        email: { label: "Email" },
        password: { label: "Senha", type: "password" },
      },
      async authorize(raw) {
        const parsed = credentialsSchema.safeParse(raw);
        if (!parsed.success) return null;

        // chama o NestJS pra validar credenciais
        const res = await fetch(`${process.env.API_URL}/auth/login`, {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify(parsed.data),
        });
        if (!res.ok) return null;
        const user = await res.json();
        return { id: user.id, email: user.email, name: user.name };
      },
    }),
  ],
  callbacks: {
    async jwt({ token, user }) {
      if (user) token.sub = user.id;
      return token;
    },
    async session({ session, token }) {
      if (token.sub) session.user.id = token.sub;
      return session;
    },
  },
  jwt: {
    // Auth.js v5 por default usa JWE (encrypted). Precisamos JWS (signed) pro NestJS validar.
    async encode({ token, secret }) {
      const { SignJWT } = await import("jose");
      const secretKey = new TextEncoder().encode(
        Array.isArray(secret) ? secret[0] : secret,
      );
      return new SignJWT(token as any)
        .setProtectedHeader({ alg: "HS256" })
        .setIssuedAt()
        .setExpirationTime("30d")
        .sign(secretKey);
    },
    async decode({ token, secret }) {
      if (!token) return null;
      const { jwtVerify } = await import("jose");
      const secretKey = new TextEncoder().encode(
        Array.isArray(secret) ? secret[0] : secret,
      );
      const { payload } = await jwtVerify(token, secretKey);
      return payload as any;
    },
  },
});
```

**Ponto crítico**: Auth.js v5 usa JWE por default (token encrypted). NestJS não decripta JWE facilmente. Trocamos o encode/decode pra usar **JWS (HS256 signed)**, que o `passport-jwt` sabe verificar nativamente.

### Cadastro (Sign up)

Auth.js não gerencia cadastro por padrão em `Credentials`. Você cria endpoint próprio no NestJS (`POST /auth/register`), Next chama direto via Server Action e depois faz `signIn`.

### Guard no NestJS

`apps/api/src/auth/jwt.strategy.ts`:

```typescript
import { Injectable } from "@nestjs/common";
import { PassportStrategy } from "@nestjs/passport";
import { Strategy, ExtractJwt } from "passport-jwt";

export type JwtPayload = {
  sub: string;
  email?: string;
  iat: number;
  exp: number;
};

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: process.env.AUTH_SECRET,
      algorithms: ["HS256"],
    });
  }

  async validate(payload: JwtPayload) {
    // O que retornar aqui vira req.user
    return { id: payload.sub, email: payload.email };
  }
}
```

`apps/api/src/auth/jwt-auth.guard.ts`:

```typescript
import { Injectable } from "@nestjs/common";
import { AuthGuard } from "@nestjs/passport";

@Injectable()
export class JwtAuthGuard extends AuthGuard("jwt") {}
```

`apps/api/src/auth/auth.module.ts`:

```typescript
import { Module } from "@nestjs/common";
import { PassportModule } from "@nestjs/passport";
import { JwtStrategy } from "./jwt.strategy";

@Module({
  imports: [PassportModule],
  providers: [JwtStrategy],
  exports: [PassportModule],
})
export class AuthModule {}
```

### Decorator @CurrentUser

Pra evitar ler `req.user` na mão em cada controller:

`apps/api/src/auth/current-user.decorator.ts`:

```typescript
import { createParamDecorator, ExecutionContext } from "@nestjs/common";

export type AuthUser = { id: string; email: string };

export const CurrentUser = createParamDecorator(
  (_: unknown, ctx: ExecutionContext): AuthUser => {
    const request = ctx.switchToHttp().getRequest();
    return request.user;
  },
);
```

Uso num controller:

```typescript
@Controller("entries")
@UseGuards(JwtAuthGuard)
export class EntriesController {
  constructor(private readonly service: EntriesService) {}

  @Get()
  list(@CurrentUser() user: AuthUser) {
    return this.service.listByUser(user.id);
  }

  @Post()
  create(@CurrentUser() user: AuthUser, @Body() body: CreateEntryDto) {
    return this.service.create(user.id, body);
  }
}
```

### Fetch wrapper no Next.js

`apps/web/src/lib/api.ts`:

```typescript
import { auth } from "@/auth";
import { cookies } from "next/headers";

const API_URL = process.env.API_URL ?? "http://localhost:3001";

// Server-side: usa auth() pra pegar o JWT do cookie
export async function apiFetch<T>(
  path: string,
  init: RequestInit = {},
): Promise<T> {
  const session = await auth();
  if (!session) throw new Error("Not authenticated");

  // Auth.js armazena o JWT no cookie. Recuperamos o valor cru.
  const cookieStore = await cookies();
  const sessionCookie = cookieStore.get("authjs.session-token")?.value;
  if (!sessionCookie) throw new Error("No session cookie");

  const res = await fetch(`${API_URL}${path}`, {
    ...init,
    headers: {
      ...init.headers,
      Authorization: `Bearer ${sessionCookie}`,
      "Content-Type": "application/json",
    },
  });

  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json();
}
```

Uso em Server Component:

```typescript
import { apiFetch } from '@/lib/api';

export default async function EntriesPage() {
  const entries = await apiFetch<Entry[]>('/entries');
  return <EntriesList entries={entries} />;
}
```

Uso em Server Action:

```typescript
"use server";
import { apiFetch } from "@/lib/api";
import { CreateEntryInput } from "@waypoint/types";

export async function createEntry(input: CreateEntryInput) {
  return apiFetch("/entries", {
    method: "POST",
    body: JSON.stringify(input),
  });
}
```

### Endpoints públicos no NestJS

Rotas como `POST /auth/login` e `POST /auth/register` **não** ficam com `@UseGuards(JwtAuthGuard)`. São chamadas pelo Auth.js antes do login existir.

Aplique o guard globalmente e libere as públicas com decorator `@Public()`, ou aplique controller-a-controller. Pra volume atual, aplicar por controller é mais simples e mais explícito.

### CORS

Se por algum momento a decisão B ficar pesada e você quiser expor NestJS diretamente ao browser em queries triviais, precisa habilitar CORS no NestJS:

```typescript
// apps/api/src/main.ts
app.enableCors({
  origin: process.env.WEB_URL,
  credentials: true,
});
```

Enquanto Next é BFF pra tudo autenticado, CORS fica desabilitado. Menos superfície pra dor de cabeça.

### Fluxo completo (do clique ao Postgres)

```
[Browser]
   ↓ POST /api/auth/callback/credentials  (Auth.js route)
[Next.js Auth.js]
   ↓ authorize() → POST /auth/login
[NestJS]
   ↓ valida senha (bcrypt vs coluna users.password_hash)
   ← 200 { id, email, name }
[Next.js Auth.js]
   ↓ jwt.encode() com HS256 e AUTH_SECRET
   ← Set-Cookie: authjs.session-token=<JWT>
[Browser]
   ↓ navega pra /dashboard
[Next.js Server Component]
   ↓ apiFetch('/entries') com Authorization: Bearer <JWT>
[NestJS JwtAuthGuard]
   ↓ passport-jwt valida com AUTH_SECRET
   ↓ req.user = { id, email }
[EntriesController.list]
   ↓ service.listByUser(user.id)
[Drizzle]
   ↓ SELECT ... WHERE user_id = $1
[Postgres]
```

## O que fica pra depois

- OAuth (Google, GitHub) via Auth.js: adicionar provider no `auth.ts`, sem mudar NestJS
- Refresh tokens: se sentir necessidade de revogação instantânea, adicionar tabela `refresh_tokens` no NestJS
- Rate limit no `/auth/login` do NestJS (`@nestjs/throttler`)
- Recuperação de senha via email (magic link ou token de reset)
- 2FA
