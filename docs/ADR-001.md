# ADR-001: Modelo polimórfico da entidade Entry

**Status:** proposto
**Data:** 2026-07-07
**Contexto:** definir estrutura de dados da entidade central do CareerOS

## Contexto

Entry é a entidade polimórfica que representa qualquer coisa capturada no Registro Rápido. Existem 8 subtipos com campos parcialmente sobrepostos:

| Subtipo         | Campos específicos                                               |
| --------------- | ---------------------------------------------------------------- |
| Aprendizado     | source, related_context                                          |
| Conquista       | achievement_type, impact_type, momento_id                        |
| Dúvida          | status (open/validate/answered), answer, answered_at, momento_id |
| Tarefa          | due_date, category, priority, is_done, completed_at              |
| Link            | url, category, reviewed                                          |
| Estudo (log)    | resource_id, minutes_spent                                       |
| Decisão técnica | project_id, tradeoffs, outcome                                   |
| Feedback        | source, feedback_type                                            |

Todos compartilham: id, user_id, title, body, tags, created_at, updated_at, type.

Existem três padrões clássicos para modelar isso em Postgres, e um híbrido que costuma vencer na prática.

## Opções consideradas

### Opção A: Single Table Inheritance (uma tabela grande)

Uma tabela `entries` com todas as colunas possíveis, a maioria nullable. Uma coluna `type` diz qual subtipo é.

**Prós:** consulta polimórfica trivial (SELECT \* FROM entries WHERE user_id = X). Sem joins. Migration simples no início.

**Contras:** tabela feia com 20+ colunas quase todas nullable. Nada impede um Link de ter due_date preenchida por engano. Índices por subtipo ficam caros porque cobrem linhas irrelevantes. Não ensina praticamente nada de SQL interessante.

**Veredito:** descartada. Escala mal e não agrega valor didático.

### Opção B: Table-Per-Type (base + tabela por subtipo)

Tabela base `entries` com campos comuns, e uma tabela específica por subtipo (`tasks`, `questions`, `achievements`, etc.) com FK 1-1 para `entries.id`.

```
entries (id, user_id, type, title, body, tags, created_at, updated_at)
├── tasks (entry_id FK, due_date, category, priority, is_done, completed_at)
├── questions (entry_id FK, status, answer, answered_at, momento_id)
├── achievements (entry_id FK, achievement_type, impact_type, momento_id)
├── links (entry_id FK, url, category, reviewed)
├── learnings (entry_id FK, source, related_context)
├── study_logs (entry_id FK, resource_id, minutes_spent)
├── decisions (entry_id FK, project_id, tradeoffs, outcome)
└── feedbacks (entry_id FK, source, feedback_type)
```

**Prós:** tipagem forte no banco. Constraints e índices por subtipo. Migrations por subtipo isoladas. Ensina joins reais e modelagem relacional adulta.

**Contras:** 9 tabelas pra manter. Query polimórfica ("dá tudo dessa semana") vira LEFT JOIN de 8 tabelas. Prisma suporta mas não é elegante. Migrations pra cada mudança de campo em qualquer subtipo.

**Veredito:** tecnicamente correta e didaticamente rica, mas custo de manutenção alto pra projeto solo de 2h/dia. Se você chegar aqui direto no Sprint 1, vai gastar Sprint 2 inteiro reclamando das queries polimórficas.

### Opção C: JSONB puro

Uma tabela `entries` com uma coluna `data JSONB` que carrega tudo específico do subtipo. Validação de forma via Zod na aplicação.

```
entries (id, user_id, type, title, body, tags, data JSONB, created_at, updated_at)
```

**Prós:** uma tabela. Adicionar campo em qualquer subtipo é mudança de código, não migration. Query polimórfica trivial.

**Contras:** banco não protege nada. Um bug no app pode salvar Task sem due_date e ninguém percebe. Índice em campo dentro de JSONB precisa ser expression index ou GIN, é mais chato de acertar. Não te ensina Postgres, te ensina a evitar Postgres.

**Veredito:** rápido pra começar, mas você tá construindo isso justamente pra aprender backend. JSONB puro faz o SQL sumir.

### Opção D (recomendada): Híbrido evolutivo

Começa com base + JSONB (opção C), e **promove subtipos pra tabela própria (opção B) quando eles ganham peso**.

Estrutura inicial:

```
entries (
  id, user_id, type, title, body,
  tags text[], data jsonb,
  created_at, updated_at
)
```

Regra de promoção: um subtipo vira tabela própria quando qualquer uma dessas coisas acontecer:

1. Você precisa de mais de dois índices sobre campos dele
2. Você tem query complexa envolvendo agregação (`SUM`, `COUNT`, `GROUP BY`)
3. Você percebe que a validação Zod não tá bastando e quer constraint no banco
4. O subtipo tem relação FK com outra entidade que você quer consultar em ambos sentidos

Aplicando essa regra aos 8 subtipos hoje, **dois nascem já em tabela própria** porque são o núcleo interativo do app:

- **`tasks`**: filtro por is_done + due_date + priority é a query principal do widget Hoje. Roda toda vez que você abre o dashboard. Merece índice composto.
- **`questions`**: filtro por status = 'open' é a query do widget Dúvidas Abertas. Também dashboard hit.

Os outros seis nascem em JSONB e podem ser promovidos depois:

- Achievement: pode promover quando você começar a agrupar por impacto/tipo pro Brag Document
- Study log: pode promover quando começar a somar minutos por trilha
- Decision: pode promover se ligar com Projects for consultado dos dois lados
- Learning, Link, Feedback: provavelmente ficam em JSONB pra sempre, são mais catálogo que domínio

## Decisão

Opção D. Começa híbrido, com `tasks` e `questions` já em tabelas próprias no Sprint 1, e o resto em JSONB. Migração pra tabela própria acontece por sprint conforme os subtipos ganharem peso.

## Consequências

**Positivas:**

- Curva de aprendizado gradual: você encosta em JSONB, GIN indexes, expression indexes E em relações 1-1, joins polimórficos, migrations não-triviais. Cobre bastante Postgres sem tudo de uma vez.
- Schema inicial minúsculo: 3 tabelas (`entries`, `tasks`, `questions`). Você entrega Sprint 1 sem sofrer.
- Promoções futuras viram exercícios didáticos legítimos ("como migrar dados de JSONB pra tabela própria sem downtime"), coisa de senior backend real.

**Negativas:**

- Zod schemas ficam obrigatórios no pacote compartilhado `packages/types` desde o primeiro dia, senão o JSONB vira lixão. Isso não é opcional.
- Query polimórfica ("dá as últimas 20 Entries de qualquer tipo") sofre um LEFT JOIN pra `tasks` e `questions`. Aceitável dado o volume esperado (uso pessoal, dezenas de linhas por dia).
- Você precisa se disciplinar pra promover subtipos quando a regra de promoção disparar. Se ignorar, JSONB cresce e vira dívida.

**Neutras:**

- Prisma modela isso bem. Base + subtipos são só relações 1-1 opcionais. JSONB é tipo `Json` e você refina com Zod. Sem necessidade de discriminated unions no ORM.

## Detalhes de implementação

Schema Prisma inicial (esboço):

```prisma
enum EntryType {
  LEARNING
  ACHIEVEMENT
  QUESTION
  TASK
  LINK
  STUDY_LOG
  DECISION
  FEEDBACK
}

model Entry {
  id        String   @id @default(cuid())
  userId    String
  type      EntryType
  title     String
  body      String?
  tags      String[]
  data      Json?     // usado pelos subtipos ainda não promovidos
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  user      User      @relation(fields: [userId], references: [id])
  task      Task?
  question  Question?

  @@index([userId, type, createdAt(sort: Desc)])
  @@index([userId, createdAt(sort: Desc)])
}

model Task {
  entryId     String   @id
  dueDate     DateTime?
  category    String
  priority    Int      @default(2)
  isDone      Boolean  @default(false)
  completedAt DateTime?

  entry       Entry    @relation(fields: [entryId], references: [id], onDelete: Cascade)

  @@index([isDone, dueDate])
}

model Question {
  entryId     String   @id
  status      String   // 'open' | 'validate' | 'answered'
  answer      String?
  answeredAt  DateTime?
  momentoId   String?

  entry       Entry    @relation(fields: [entryId], references: [id], onDelete: Cascade)

  @@index([status])
}
```

Zod schemas compartilhados em `packages/types/entry.ts`:

```typescript
export const LearningDataSchema = z.object({
  source: z.string().optional(),
  relatedContext: z.string().optional(),
});

export const LinkDataSchema = z.object({
  url: z.string().url(),
  category: z.string().optional(),
  reviewed: z.boolean().default(false),
});

// ...um por subtipo em JSONB

export const CreateEntryInput = z.discriminatedUnion("type", [
  z.object({
    type: z.literal("LEARNING"),
    title: z.string(),
    body: z.string().optional(),
    tags: z.array(z.string()),
    data: LearningDataSchema,
  }),
  z.object({
    type: z.literal("LINK"),
    title: z.string(),
    body: z.string().optional(),
    tags: z.array(z.string()),
    data: LinkDataSchema,
  }),
  // ...
  z.object({
    type: z.literal("TASK"),
    title: z.string(),
    body: z.string().optional(),
    tags: z.array(z.string()),
    task: TaskInputSchema,
  }),
  z.object({
    type: z.literal("QUESTION"),
    title: z.string(),
    body: z.string().optional(),
    tags: z.array(z.string()),
    question: QuestionInputSchema,
  }),
]);
```

Full-text search: crie coluna gerada `tsv` em `entries` com `to_tsvector('portuguese', title || ' ' || coalesce(body, ''))` e índice GIN. Cobre 90% das buscas. Pra buscar dentro do JSONB específico, adicione expression index quando precisar.
