# ADR-003: Estrutura de monorepo

**Status:** proposto
**Data:** 2026-07-07
**Contexto:** definir como organizar backend, frontend e pacotes compartilhados no mesmo repo

## Contexto

O Waypoint tem dois apps (`apps/api` em NestJS e `apps/web` em Next.js) que compartilham código relevante: Zod schemas de validação, tipos derivados do schema Drizzle, e provavelmente um design system mínimo (`packages/ui`). Manter isso em repos separados significa duplicar tipos, ou publicar packages internos em registry privado. Nenhuma das duas é aceitável pra projeto solo de 2h/dia.

Monorepo já foi decidido. A questão agora é **qual ferramenta gerencia o monorepo** e qual estrutura interna adotar.

## Opções consideradas

### Opção A: pnpm workspaces puro

Só pnpm + `pnpm-workspace.yaml` + TypeScript project references. Sem camada adicional.

**Prós:**

- Zero configuração além do `pnpm-workspace.yaml`
- Symlinks reais entre packages via `workspace:*`, sem publicação
- pnpm sozinho já resolve dedupe, install rápido, node_modules nested (mais correto que hoisted)
- Você aprende **uma** ferramenta, não duas empilhadas
- Sem magia, tudo transparente

**Contras:**

- Sem build cache. Roda `pnpm build` no monorepo, ele recompila tudo
- Sem grafo de tasks (você orquestra na mão via `pnpm -r` ou `pnpm --filter`)
- Sem paralelização inteligente entre tarefas dependentes

### Opção B: Turborepo em cima de pnpm workspaces

Adiciona `turbo.json` para orquestrar tasks, cachear builds e paralelizar. Vercel é dona, integra bem com Next.js.

**Prós:**

- Cache local (e remoto, opcional) de builds. Roda o mesmo `build` duas vezes, a segunda é instantânea
- Task graph automático: `turbo build` sabe que `apps/web` depende de `packages/ui` e builda na ordem certa
- Paralelização entre tasks independentes
- `turbo dev` sobe todos os apps ao mesmo tempo com um comando

**Contras:**

- Camada a mais. Você mantém `turbo.json` E `pnpm-workspace.yaml`
- Configurar caching keys corretamente exige atenção. Cache errado gera bug fantasma
- Overhead que só compensa quando builds ficam lentos, o que não é seu caso nos primeiros meses

### Opção C: Nx

Framework robusto de monorepo com generators, plugins, dependency graph visual, cache distribuído.

**Prós:**

- Melhor DX pra monorepo grande. Comandos `nx g @nx/nest:app api` que geram estrutura completa
- Grafo visual das dependências
- Cache distribuído nativo
- Suporte forte pra NestJS e Next.js oficialmente

**Contras:**

- Opinions fortes. Ele quer gerenciar tudo do jeito dele
- Curva íngreme. Você aprende Nx **antes** de conseguir escrever a primeira feature
- Feito para times grandes com muitos apps. Você tem dois. E vai continuar tendo dois
- Substitui o `nest g` do NestJS por generators do Nx, o que atrapalha o objetivo de aprender NestJS puro

**Veredito:** exagero pro contexto. Fica anotado como referência caso o projeto vire algo com múltiplos apps.

## Decisão

**pnpm workspaces puro (Opção A).**

Turborepo entra depois **se e quando** você sentir dor real de build. Nx fica descartado.

Motivos:

1. **Você tem dois apps.** Task graph pra dois apps não justifica ferramenta de orquestração. `pnpm -r --parallel dev` sobe backend e frontend juntos e resolve.

2. **Você não faz build no CI o tempo todo.** Side project noturno não pressiona pipeline. O ganho de cache do Turborepo só aparece quando build vira gargalo, e enquanto for 2h/dia num app pequeno, isso demora.

3. **Uma ferramenta a menos pra aprender.** Você tá aprendendo NestJS, Drizzle em contexto novo, integração NestJS + Next via JWT. Adicionar Turborepo agora dilui foco sem retorno claro.

4. **Migração pra Turborepo depois é barata.** Adicionar `turbo.json` e mudar scripts pra `turbo dev` em vez de `pnpm dev` leva uma hora. Se um dia doer, você resolve num sábado.

## Consequências

**Positivas:**

- Setup do Sprint 0 fica mais leve. Menos config, mais código.
- Symlinks reais entre packages significam que você muda `packages/types` e o `apps/api` vê a mudança na hora, sem rebuild manual.
- Você aprende monorepo pelo caminho fundamental (workspaces + TypeScript project references), o que é conhecimento transferível.

**Negativas:**

- Se um dia adicionar `packages/emails`, `packages/config`, `packages/scripts`, etc., a orquestração manual começa a incomodar. Aí é hora de olhar Turborepo.
- Sem cache incremental, CI vai rodar todos os checks sempre. Aceitável pro volume atual.
- Você monta o setup Vercel manualmente se for deployar via Vercel (rootDirectory + buildCommand precisam apontar pra `apps/web` com contexto do monorepo).

**Neutras:**

- pnpm exige `.npmrc` com algumas configurações. Documentado abaixo.
- Node modules ficam nested, o que é mais correto mas ocasionalmente confunde ferramentas que assumem hoisting (raro em 2026).

### `pnpm-workspace.yaml`

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

### `.npmrc` na raiz

```
link-workspace-packages=true
prefer-workspace-packages=true
save-workspace-protocol=rolling
strict-peer-dependencies=false
```

- `link-workspace-packages=true`: automaticamente linka packages internos quando um app declara dependência
- `prefer-workspace-packages=true`: preferir versão do workspace sobre registry
- `save-workspace-protocol=rolling`: quando você `pnpm add`, salva como `workspace:*` (não fixa versão)
- `strict-peer-dependencies=false`: Next.js e alguns pacotes reclamam peer deps sem esse override

### `package.json` da raiz

```json
{
  "name": "waypoint",
  "private": true,
  "packageManager": "pnpm@9.15.0",
  "scripts": {
    "dev": "pnpm -r --parallel dev",
    "build": "pnpm -r build",
    "lint": "pnpm -r lint",
    "typecheck": "pnpm -r typecheck",
    "test": "pnpm -r test",
    "db:generate": "pnpm --filter @waypoint/api db:generate",
    "db:migrate": "pnpm --filter @waypoint/api db:migrate",
    "db:studio": "pnpm --filter @waypoint/api db:studio"
  },
  "devDependencies": {
    "prettier": "^3.4.0",
    "typescript": "^5.7.0"
  }
}
```

`pnpm -r` roda o script em todos os packages que definem ele. `--parallel` roda ao mesmo tempo. `--filter @waypoint/api` roda só no pacote nomeado.

### `packages/types/package.json`

```json
{
  "name": "@waypoint/types",
  "version": "0.0.0",
  "private": true,
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "scripts": {
    "typecheck": "tsc --noEmit",
    "lint": "eslint src"
  },
  "dependencies": {
    "zod": "^3.24.0"
  }
}
```

Note que `main` aponta pra `.ts`, não `.js`. Isso funciona porque tanto NestJS quanto Next.js compilam TypeScript em tempo real. Você não publica esse pacote em registry, ele só existe internamente. Zero build step para packages internos até que precise.

### `apps/api/package.json` (trecho relevante)

```json
{
  "name": "@waypoint/api",
  "dependencies": {
    "@waypoint/types": "workspace:*",
    "@nestjs/core": "^10.4.0",
    "drizzle-orm": "^0.36.0",
    "pg": "^8.13.0",
    "zod": "^3.24.0"
  }
}
```

O `workspace:*` diz "pega do workspace, qualquer versão". pnpm resolve automaticamente pro `packages/types`.

### `apps/web/package.json` (trecho relevante)

```json
{
  "name": "@waypoint/web",
  "dependencies": {
    "@waypoint/types": "workspace:*",
    "@waypoint/ui": "workspace:*",
    "next": "^15.1.0",
    "react": "^19.0.0",
    "zod": "^3.24.0"
  }
}
```

### TypeScript project references

Cada `tsconfig.json` aponta pros packages que consome. Isso ajuda IDE (VSCode) a navegar corretamente e o tsc a validar em ordem.

`apps/api/tsconfig.json`:

```json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "references": [{ "path": "../../packages/types" }],
  "include": ["src/**/*"]
}
```

`packages/types/tsconfig.json`:

```json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "composite": true,
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src/**/*"]
}
```

`tsconfig.base.json` na raiz com opções compartilhadas (`strict`, `target`, `moduleResolution`, etc.).

### Fluxo de desenvolvimento típico

Sobe tudo com um comando:

```bash
pnpm dev
```

Sobe backend em `localhost:3001`, frontend em `localhost:3000`. Você mexe em `packages/types/src/entry.ts` e o TypeScript no `apps/web` já sabe do novo tipo imediatamente, sem restart.

Adicionar dependência em app específico:

```bash
pnpm --filter @waypoint/api add express-rate-limit
pnpm --filter @waypoint/web add lucide-react
```

Adicionar dependência na raiz (dev tool compartilhada):

```bash
pnpm add -D -w prettier-plugin-tailwindcss
```

O `-w` significa "workspace root".

### CI (GitHub Actions)

`.github/workflows/ci.yml`:

```yaml
name: CI
on: [push, pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: "pnpm"
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck
      - run: pnpm test
```

Sem cache incremental (é a limitação de pnpm puro), mas com `actions/setup-node` cacheando `~/.pnpm-store` a instalação fica rápida.

### Deploy (nota importante)

Se um dia deployar via Vercel, configure o projeto apontando `rootDirectory` pra `apps/web` e `installCommand` pra `pnpm install --frozen-lockfile` executando na raiz. Vercel tem preset "pnpm monorepo" nativo, funciona bem.

Backend NestJS não vai na Vercel (não é serverless-friendly sem adaptação). Railway, Fly.io ou Render, com Docker build que faz `pnpm --filter @waypoint/api... build` funciona sem drama.
