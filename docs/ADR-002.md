# ADR-002: Escolha de ORM para o backend NestJS

**Status:** proposto
**Data:** 2026-07-07
**Contexto:** definir camada de acesso a dados no NestJS + Postgres
**Substitui:** o esboço de schema Prisma do ADR-001, que era placeholder

## Contexto

O backend do CareerOS é NestJS + Postgres 16. Precisa suportar:

- Modelo híbrido do ADR-001 (base `entries` + subtipos `tasks`/`questions` promovidos, resto em JSONB)
- Full-text search em Postgres via tsvector + GIN
- Zod schemas compartilhados em `packages/types` para validar JSONB
- Migrations rastreáveis em git
- Boa DX pra side project noturno (2h/dia, sem paciência pra ORM complicado)

Objetivo secundário importante: você quer **aprender SQL/Postgres deeply**. Isso vira critério de decisão, não só nota de rodapé. Um ORM que abstrai SQL demais te tira o que você quer treinar.

## Opções consideradas

### Opção A: Prisma

O padrão moderno para Node.js. Schema em DSL própria (`schema.prisma`), cliente TypeScript gerado, Prisma Studio pra explorar dados visualmente.

**Prós:**

- DX excelente, autocomplete forte, mensagens de erro claras
- Prisma Studio (interface visual pro banco) é ótimo pra você que é visual
- Migrations declarativas (muda o schema, roda `prisma migrate dev`)
- Documentação de referência mundial
- Ecossistema NestJS bem estabelecido (guias oficiais)

**Contras:**

- Abstrai bastante o SQL. Pra queries complexas você recorre a `$queryRaw`, o que quebra a experiência
- Schema em DSL não-TypeScript. Você escreve `.prisma`, não `.ts`
- Code generation obrigatório (`prisma generate` toda vez que mexe no schema)
- Full-text search e GIN indexes precisam ser adicionados via SQL puro nas migrations, meio fora do fluxo declarativo
- Pra o objetivo de aprender Postgres, tende a esconder o que você quer ver

### Opção B: TypeORM

O ORM histórico do ecossistema NestJS. Entidades como classes TypeScript com decorators.

**Prós:**

- Integração NestJS mais antiga e testada
- `nest g resource` gera CRUD completo incluindo entity, service, controller e DTO
- Decorators combinam com o estilo do resto do NestJS
- Suporta tanto Active Record quanto Data Mapper

**Contras:**

- Manutenção morna nos últimos anos. Vários bugs conhecidos que se arrastam
- Tipagem tem pontos fracos, especialmente em relações e queries complexas
- Decorators pesados poluem as entidades
- Migrations menos fluidas que Prisma ou Drizzle
- Comunidade se movendo pra Prisma e Drizzle

### Opção C: Drizzle

SQL-first. Você escreve schema em TypeScript puro usando funções que espelham SQL. Query builder tipado. Sem code generation.

**Prós:**

- Schema é TypeScript puro, sem DSL separada
- Query builder desenhado pra parecer SQL. Você continua vendo o que o banco vai executar
- Migrations em SQL nativo, versionadas em git. Você lê e entende cada uma
- Suporte de primeira classe pra JSONB com `$type<>()` que tipa o conteúdo
- Full-text search, GIN indexes, expression indexes: tudo acessível diretamente
- Drizzle Studio existe (menos poderoso que Prisma Studio, mas cobre o essencial)
- **Você já usa no FinSight AI**. Reduz uma variável nova quando você tá aprendendo NestJS ao mesmo tempo

**Contras:**

- Ecossistema NestJS menor. Sem `nest g resource` equivalente. Você configura injeção manualmente
- Documentação boa mas menos densa que Prisma
- Menos "guias oficiais" pra padrões de produção. Você vai improvisar mais

### Opção D: Kysely (query builder puro)

Query builder tipado sem ORM. Migrations gerenciadas por fora.

**Prós:** o mais fino, mais próximo do SQL. Tipagem excelente.

**Contras:** não é ORM, você monta a camada de repositório na mão. Pra projeto solo à noite, é overhead sem retorno claro.

**Veredito:** interessante mas não pra agora. Fica anotado como opção pra um projeto futuro focado em performance.

## Decisão

**Drizzle.**

O peso decisivo é o alinhamento com os dois objetivos que você declarou:

1. **Aprender Postgres deeply.** Drizzle é o único das opções que te mantém em contato constante com o SQL real. Você vê o join, você escreve o índice GIN na migration, você entende o que a coluna gerada tsvector faz. Prisma resolveria isso pra você e você aprenderia Prisma, não Postgres.

2. **Reduzir superfície de aprendizado paralelo.** Você já tá encarando NestJS pela primeira vez em uso próprio. Adicionar Prisma junto significa duas ferramentas novas simultâneas. Drizzle é a única onde a sintaxe já vai estar na sua memória do FinSight AI. Isso libera tempo mental pra focar no NestJS (guards, interceptors, pipes, providers), que é o que você realmente quer treinar.

O trade-off é real: você abre mão do Prisma Studio (que é bom pra você que é visual) e da geração automática de CRUD do TypeORM. Compensações:

- Drizzle Studio (`drizzle-kit studio`) cobre a maior parte do que você precisa visualmente
- Ausência do `nest g resource` é oportunidade de aprender a estrutura de módulos do NestJS na mão, o que é bom didaticamente

## Consequências

**Positivas:**

- Schema, migrations e queries em TypeScript puro. Um único vocabulário.
- Você mantém contato com SQL. Full-text search em Portuguese, GIN em JSONB, expression indexes ficam todos legíveis no seu código.
- Reaproveita a fluência que você já tem do FinSight AI.
- Zero code generation. Menos fricção no ciclo dev.

**Negativas:**

- Você monta a integração NestJS na mão. Vai criar um `DrizzleModule` global que expõe o cliente via injeção. Duas horas de setup no Sprint 0.
- Sem Prisma Studio. Se sentir falta forte, roda `drizzle-kit studio` ou usa TablePlus/Beekeeper Studio como cliente Postgres.
- Se um dia quiser mudar pra Prisma, o custo não é trivial (schema DSL, code generation). Mas isso é problema hipotético.

**Neutras:**

- Zod continua sendo a validação nas bordas (controllers) e nos `$type<>()` do JSONB. Nada muda no ADR-001 aqui.
- Migrations viram SQL na pasta `drizzle/`. Podem ser revisadas em PR como código.

### Schema do ADR-001 refeito em Drizzle

Substitui o exemplo Prisma do ADR-001. Este é o oficial:

```typescript
// apps/api/src/db/schema/entries.ts
import {
  pgTable,
  text,
  timestamp,
  jsonb,
  index,
  pgEnum,
} from "drizzle-orm/pg-core";
import { createId } from "@paralleldrive/cuid2";
import { users } from "./users";

export const entryTypeEnum = pgEnum("entry_type", [
  "learning",
  "achievement",
  "question",
  "task",
  "link",
  "study_log",
  "decision",
  "feedback",
]);

// Tipo do payload JSONB por subtipo, validado por Zod no controller
export type EntryData =
  | { source?: string; relatedContext?: string } // learning
  | { url: string; category?: string; reviewed: boolean } // link
  | { achievementType: string; impactType: string; momentoId?: string } // achievement
  | { resourceId: string; minutesSpent: number } // study_log
  | { projectId?: string; tradeoffs: string; outcome?: string } // decision
  | { source: string; feedbackType: "positive" | "constructive" }; // feedback

export const entries = pgTable(
  "entries",
  {
    id: text("id")
      .primaryKey()
      .$defaultFn(() => createId()),
    userId: text("user_id")
      .notNull()
      .references(() => users.id, { onDelete: "cascade" }),
    type: entryTypeEnum("type").notNull(),
    title: text("title").notNull(),
    body: text("body"),
    tags: text("tags").array().notNull().default([]),
    data: jsonb("data").$type<EntryData | null>(),
    createdAt: timestamp("created_at").defaultNow().notNull(),
    updatedAt: timestamp("updated_at").defaultNow().notNull(),
  },
  (t) => ({
    userTypeDate: index("idx_entries_user_type_date").on(
      t.userId,
      t.type,
      t.createdAt,
    ),
    userDate: index("idx_entries_user_date").on(t.userId, t.createdAt),
  }),
);
```

### Full-text search: coluna gerada + índice GIN

Isso é migration SQL manual, não gerada por Drizzle. Fica em `drizzle/000X_fts.sql`:

```sql
ALTER TABLE entries
  ADD COLUMN tsv tsvector
  GENERATED ALWAYS AS (
    to_tsvector('portuguese', coalesce(title, '') || ' ' || coalesce(body, ''))
  ) STORED;

CREATE INDEX idx_entries_tsv ON entries USING GIN (tsv);
```

Query no service:

```typescript
import { sql } from 'drizzle-orm';

async search(userId: string, query: string) {
  return this.db
    .select()
    .from(entries)
    .where(sql`user_id = ${userId} AND tsv @@ plainto_tsquery('portuguese', ${query})`)
    .orderBy(sql`ts_rank(tsv, plainto_tsquery('portuguese', ${query})) DESC`)
    .limit(20);
}
```

### DrizzleModule no NestJS (esqueleto)

```typescript
// apps/api/src/db/drizzle.module.ts
import { Global, Module } from "@nestjs/common";
import { drizzle } from "drizzle-orm/node-postgres";
import { Pool } from "pg";
import * as schema from "./schema";

export const DRIZZLE = Symbol("DRIZZLE");

@Global()
@Module({
  providers: [
    {
      provide: DRIZZLE,
      useFactory: () => {
        const pool = new Pool({ connectionString: process.env.DATABASE_URL });
        return drizzle(pool, { schema });
      },
    },
  ],
  exports: [DRIZZLE],
})
export class DrizzleModule {}
```

Uso num service:

```typescript
@Injectable()
export class EntriesService {
  constructor(@Inject(DRIZZLE) private db: NodePgDatabase<typeof schema>) {}

  async listByUser(userId: string) {
    return this.db.select().from(entries).where(eq(entries.userId, userId));
  }
}
```
